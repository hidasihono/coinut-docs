## Overview

Coinut sends webhook events to your registered callback URL when important events occur.

All webhooks:

- Use the `POST` method
- Send a JSON body using the `{event, payload}` envelope shown in [Webhook Payload Examples](webhooks-payload-examples)
- Are only sent for event types you've subscribed to, and only once you have a callback URL configured (see [Setting Up Webhooks](#setting-up-webhooks))
- Are signed with the same HMAC-SHA256 scheme used for incoming requests (see [Webhook Signature Verification](webhooks-signature-verification))
- Accept any `2xx` response as a successful delivery — respond as soon as you've durably queued the event, since no delivery timeout is currently enforced (see [Delivery & Reliability](#delivery-reliability))

## Setting Up Webhooks

<Steps>
  <Step title="Register callback URL">
    Send [`POST /callback-url`](api/common-callback-url-update) with your HTTPS endpoint.

    ```json
    {
      "callbackUrl": "https://your-domain.com/webhooks/coinut"
    }
    ```
  </Step>
  <Step title="Subscribe to events">
    Send [`POST /subscribed-events`](api/common-subscribed-events-update) with the event types you want to receive.

    ```json
    {
      "events": [
        "CUSTOMER_APPROVED",
        "DEPOSIT_RECEIVED",
        "TRADE_SETTLED"
      ]
    }
    ```

    <Note>
      Both steps are required — an event is only queued for delivery if you have a callback URL **and** that event type appears in your subscribed events. Missing either one means the event is silently skipped rather than retried later.
    </Note>
  </Step>
  <Step title="Verify signature">
    Every webhook is signed exactly like an incoming API request (see [Authentication](authentication)): the same four headers (`X-API-Key`, `X-Timestamp`, `X-Nonce`, `X-Signature`) are sent with the callback, built from the same HMAC-SHA256 canonical string. Verify it before trusting the payload — see [Webhook Signature Verification](webhooks-signature-verification) for the exact construction.
  </Step>
  <Step title="Acknowledge receipt">
    Return any `2xx` status as soon as you've durably stored or queued the event. There is currently no server-side delivery timeout, but you should still respond quickly and process asynchronously — a slow handler ties up a delivery worker rather than protecting you from anything. Failed deliveries are retried up to 5 times with increasing backoff (10 minutes, 1 hour, 1 day, 1 week), so your handler must be idempotent and safe to receive duplicate deliveries — see [Delivery & Reliability](#delivery-reliability).
  </Step>
</Steps>

## Event Types

| Event | Description | Payload Type |
| --- | --- | --- |
| `CUSTOMER_APPROVED` | Customer KYC approved | `GetCustomerDataDto` |
| `CUSTOMER_REJECTED` | Customer KYC rejected | `GetCustomerDataDto` |
| `CUSTOMER_DELETED` | Customer deleted | `GetCustomerDataDto` |
| `VIRTUAL_ACCOUNT_APPROVED` | VA created and verified | `VirtualAccountDetailDto` |
| `VIRTUAL_ACCOUNT_REJECTED` | VA creation rejected | `VirtualAccountDetailDto` |
| `DEPOSIT_ADDRESS_APPROVED` | Deposit address verified | `DepositAddressDetailDto` |
| `DEPOSIT_ADDRESS_REJECTED` | Deposit address rejected | `DepositAddressDetailDto` |
| `DEPOSIT_RECEIVED` | Deposit detected | `PartnerDepositDetailDataDto` |
| `DEPOSIT_APPROVED` | Deposit confirmed | `PartnerDepositDetailDataDto` |
| `DEPOSIT_REJECTED` | Deposit rejected/refunded | `PartnerDepositDetailDataDto` |
| `TRADE_CREATED` | Trade initiated | `TradeDetailDataDto` |
| `TRADE_SETTLED` | Trade completed | `TradeDetailDataDto` |
| `PAYMENT_SETTLED` | Payment completed | `GetPaymentDetailDataDto` |

Each event type is also documented in the `WebhookEvent` schema in the [OpenAPI spec](/api-overview) (`components.schemas.WebhookEvent`), which lists every valid `event` enum value alongside the deprecated legacy names. See [Webhook Payload Examples](webhooks-payload-examples) for a full example payload of every event above.

### Legacy events

These dot-notation events are still delivered in production for backward compatibility, but are deprecated — build new integrations against the events table above instead. They use different, older payload shapes (`WebhookUserPayload`, `WebhookVaPayload`, `WebhookDepositPayload`, `WebhookTradePayload`).

| Legacy event | Replaced by |
| --- | --- |
| `user.approved` | `CUSTOMER_APPROVED` |
| `user.rejected` | `CUSTOMER_REJECTED` |
| `va.approved` | `VIRTUAL_ACCOUNT_APPROVED` |
| `va.rejected` | `VIRTUAL_ACCOUNT_REJECTED` |
| `deposit.received` | `DEPOSIT_RECEIVED` |
| `deposit.approved` | `DEPOSIT_APPROVED` |
| `deposit.rejected` | `DEPOSIT_REJECTED` |
| `trade.created` | `TRADE_CREATED` |
| `trade.settled` | `TRADE_SETTLED` |

## Delivery & Reliability

| Property | Value |
| --- | --- |
| Method | `POST` |
| Protocol | HTTPS strongly recommended. Note: the callback-URL field is not currently restricted to `https://` by validation, so a plain HTTP URL will be accepted — use HTTPS regardless, since payloads include customer and transaction data. |
| Timeout | Not enforced by the sender today — the delivery attempt will wait indefinitely for your response. Respond promptly anyway; don't rely on this. |
| Retries | Up to 5 total attempts per event occurrence |
| Backoff | 10 minutes → 1 hour → 1 day → 1 week between attempts |
| Success criteria | Any `2xx` response |
| Deduplication | Keyed on partner + event type + source record ID (surfaced to you as `payload.id`) |

<Note>
  Deduplication is scoped per partner, so it only suppresses repeat deliveries of the *same* underlying occurrence to *you*. If all 5 attempts for an occurrence are exhausted without a `2xx`, and the same underlying event fires again upstream (e.g. a corrected retry), a new delivery cycle starts for it — so `event` + `payload.id` alone isn't a lifetime guarantee against redelivery beyond the 5-attempt window. Keep your handler idempotent regardless.
</Note>

## Best Practices

- **Always verify signatures.** Validate every delivery against the [Webhook Signature Verification](webhooks-signature-verification) scheme before processing the payload — don't rely on network-level trust (e.g. IP allowlisting) alone.
- **Handle idempotently.** See [Deduplication](#delivery-reliability) — dedup is best-effort, not a lifetime guarantee, so your handler must tolerate duplicate deliveries safely.
- **Respond quickly, process asynchronously.** No delivery timeout is enforced today, but don't depend on that — acknowledge with a `2xx` as soon as the event is durably queued.
- **Log everything.** Store raw payloads and headers for debugging.
- **Expect retries.** The same event occurrence can be redelivered up to 5 times over the course of a week if your endpoint doesn't return `2xx`.

## Testing Webhooks Locally

<Tip>
  Test webhooks locally with [ngrok](https://ngrok.com). Run `ngrok http 3000`, then register the HTTPS tunnel URL as your callback endpoint. Coinut will deliver events to your local machine.
</Tip>
