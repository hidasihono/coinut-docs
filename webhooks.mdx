## Overview

Coinut sends webhook events to your registered callback URL when important events occur.

All webhooks:

- Use the `POST` method
- Send a JSON body
- Accept any `2xx` response as a successful delivery â€” respond as soon as you've durably queued the event, since no delivery timeout is currently enforced (see [Webhook Delivery](#webhook-delivery))
- Use the `{event, payload}` envelope shown in the examples below
- Are signed with the same HMAC-SHA256 scheme used for incoming requests (see [Signature Verification](#signature-verification))

## Setting Up Webhooks

<Steps>
  <Step title="Register callback URL">
    Send `POST /callback-url` with your HTTPS endpoint.

    ```json
    {
      "callbackUrl": "https://your-domain.com/webhooks/coinut"
    }
    ```
  </Step>
  <Step title="Subscribe to events">
    Send `POST /subscribed-events` with the event types you want to receive.

    ```json
    {
      "events": [
        "CUSTOMER_APPROVED",
        "DEPOSIT_RECEIVED",
        "TRADE_SETTLED"
      ]
    }
    ```
  </Step>
  <Step title="Verify signature">
    Every webhook is signed exactly like an incoming API request (see [Authentication](authentication)): the same four headers (`X-API-Key`, `X-Timestamp`, `X-Nonce`, `X-Signature`) are sent with the callback, built from the same HMAC-SHA256 canonical string. Verify it before trusting the payload â€” see [Signature Verification](#signature-verification) below for the exact construction.
  </Step>
  <Step title="Acknowledge receipt">
    Return any `2xx` status as soon as you've durably stored or queued the event. There is currently no server-side delivery timeout, but you should still respond quickly and process asynchronously â€” a slow handler ties up a delivery worker rather than protecting you from anything. Failed deliveries are retried up to 5 times with increasing backoff (10 minutes, 1 hour, 1 day, 1 week), so your handler must be idempotent and safe to receive duplicate deliveries.
  </Step>
</Steps>

## Webhook Delivery

| Property | Value |
| --- | --- |
| Method | `POST` |
| Protocol | HTTPS strongly recommended. Note: the callback-URL field is not currently restricted to `https://` by validation, so a plain HTTP URL will be accepted â€” use HTTPS regardless, since payloads include customer and transaction data. |
| Timeout | Not enforced by the sender today â€” the delivery attempt will wait indefinitely for your response. Respond promptly anyway; don't rely on this. |
| Retries | Up to 5 total attempts per event |
| Backoff | 10 minutes â†’ 1 hour â†’ 1 day â†’ 1 week between attempts |
| Success criteria | Any `2xx` response |
| Deduplication | `event` \+ `payload.id` combination |

### Legacy events

These dot-notation events are still delivered in production for backward compatibility, but are deprecated â€” build new integrations against the events table above instead. They use different, older payload shapes (`WebhookUserPayload`, `WebhookVaPayload`, `WebhookDepositPayload`, `WebhookTradePayload`).

| Legacy event | Replaced by |
| --- | --- |
| `user.approved` | `CUSTOMER_APPROVED` |
| `user.rejected` | `CUSTOMER_REJECTED` |
| `va.approved` | `VIRTUAL_ACCOUNT_APPROVED` |
| `va.rejected` | `VIRTUAL_ACCOUNT_REJECTED` |
| `deposit.received` | `DEPOSIT_RECEIVED` |
| `deposit.approved` | `DEPOSIT_APPROVED` |

## Signature Verification

Every webhook delivery is signed with the same HMAC-SHA256 scheme documented on the [Authentication](authentication) page for incoming requests â€” using your API key/secret pair, not a separate webhook secret. The request carries the same four headers:

| Header | Description |
| --- | --- |
| `X-API-Key` | Your API key |
| `X-Timestamp` | Unix timestamp in seconds, at send time |
| `X-Nonce` | A random string, unique per delivery |
| `X-Signature` | HMAC-SHA256 hex digest of the canonical string below |

Build the same newline-delimited canonical string used for request signing, then verify it with HMAC-SHA256 using your API secret:

```text
POST
<your host>
<your webhook path>
                          <- empty query-string segment
sha256(raw request body)
<X-Timestamp value>
<X-Nonce value>
```

<CodeGroup>

```javascript Node.js
const crypto = require('crypto');

function verifyWebhookSignature(req, apiSecret) {
  const { 'x-timestamp': timestamp, 'x-nonce': nonce, 'x-signature': signature } = req.headers;
  const host = req.headers.host;
  const path = req.path; // e.g. '/webhooks/coinut'
  const bodyHash = crypto.createHash('sha256').update(JSON.stringify(req.body)).digest('hex');

  const canonical = ['POST', host, path, '', bodyHash, timestamp, nonce].join('\n');
  const expected = crypto.createHmac('sha256', apiSecret).update(canonical).digest('hex');

  return crypto.timingSafeEqual(Buffer.from(expected), Buffer.from(signature));
}
```

```python Python
import hmac
import hashlib
import json

def verify_webhook_signature(headers, body: dict, path: str, api_secret: str) -> bool:
    timestamp = headers["X-Timestamp"]
    nonce = headers["X-Nonce"]
    signature = headers["X-Signature"]
    host = headers["Host"]

    body_hash = hashlib.sha256(json.dumps(body, separators=(",", ":")).encode()).hexdigest()
    canonical = "\n".join(["POST", host, path, "", body_hash, timestamp, nonce])
    expected = hmac.new(api_secret.encode(), canonical.encode(), hashlib.sha256).hexdigest()

    return hmac.compare_digest(expected, signature)
```

</CodeGroup>

<Warning>
  The exact JSON serialization of the body must match byte-for-byte on both sides for the body hash to match â€” verify against the **raw request body bytes** your framework received, not a value you've re-serialized after parsing, since key ordering or whitespace differences will produce a different hash.
</Warning>

## Best Practices

- **Always verify signatures.** Validate every delivery with the scheme above before processing the payload â€” don't rely on network-level trust (e.g. IP allowlisting) alone.
- **Handle idempotently.** Use the `event` \+ `payload.id` combination to prevent duplicate processing.
- **Respond quickly, process asynchronously.** No delivery timeout is enforced today, but don't depend on that â€” acknowledge with a `2xx` as soon as the event is durably queued.
- **Log everything.** Store raw payloads and headers for debugging.
- **Expect retries.** The same event can be redelivered up to 5 times over the course of a week if your endpoint doesn't return `2xx`.

<Tip>
  Test webhooks locally with [ngrok](https://ngrok.com). Run `ngrok http 3000`, then register the HTTPS tunnel URL as your callback endpoint. Coinut will deliver events to your local machine.
</Tip>
